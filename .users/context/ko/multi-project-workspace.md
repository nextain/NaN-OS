# 멀티 프로젝트 워크스페이스 가이드

> 미러: `.agents/context/multi-project-workspace.yaml`

## 목적

하나의 워크스페이스(예: `~/dev/`)에서 여러 프로젝트를 동시에 관리하는 메인테이너/파워 유저를 위한 가이드. 단일 레포만 클론하는 기여자는 이 가이드가 필요 없습니다.

---

## 컨텍스트 계층화

프로젝트 레벨 컨텍스트가 워크스페이스 레벨보다 우선합니다.

**해석 순서:**
1. 프로젝트 진입점 (예: `naia-os/AGENTS.md`)
2. 프로젝트 `.agents/` 컨텍스트 파일
3. 워크스페이스 진입점 (예: `~/dev/CLAUDE.md`)
4. 워크스페이스 `.agents/` 컨텍스트 파일

---

## 이름 충돌 주의

워크스페이스와 프로젝트에 같은 이름의 파일이 다른 내용으로 존재할 수 있습니다.

**알려진 사례:** `development-cycle.yaml`
- 워크스페이스 루트: 27줄 (경량, 오타/설정용)
- 프로젝트 (naia-os): 217줄 (전체 코딩 사이클 + TDD + 보안 리뷰)
- **해결**: 코딩 작업 시 프로젝트 내부 버전 사용.

---

## 작업 로그 전략

**단일 프로젝트** (기본값): 프로젝트 루트의 `work-logs/` (gitignored), `{username}/` 하위 디렉토리.

**멀티 프로젝트**: 두 가지 옵션:
1. **프로젝트별** (권장): 각 프로젝트의 `work-logs/` 디렉토리. AI가 자연스럽게 발견.
2. **중앙 집중**: 별도 레포 (예: `docs-work-logs/`). 프로젝트 간 가시성, 단 워크스페이스 경로 의존.

---

## 파일 잠금 프로토콜

같은 워크스페이스에서 여러 AI 세션을 병렬 실행할 때 사용. **단일 프로젝트 기여자는 불필요.**

- **잠금 파일**: `.claude/file-locks.json` (워크스페이스 루트 상대 경로)
- 편집 전: 다른 세션이 잠근 파일인지 확인
- 잠금 시 owner = 브랜치명
- 완료 후 해제
- `free` 배열의 파일은 자유 수정 가능
